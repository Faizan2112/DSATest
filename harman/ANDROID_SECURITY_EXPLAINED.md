# Android Security: Deep Dive & Trade-offs (Staff Level)

For an 8+ YOE interview at Harman (building Android Automotive OS or IoT), security isn't a checklist; it's a series of architectural compromises. Every security feature introduces latency, complexity, or UX friction. You must understand the **Trade-offs**.

---

## 1. Data at Rest (Storage Security)

### A. EncryptedSharedPreferences (Jetpack Security)
**How it works in detail:** 
It doesn't just encrypt the whole file. It encrypts Keys with a deterministic algorithm (AES256-SIV) so they can be looked up quickly, and encrypts Values with varying nonces (AES256-GCM) so the exact same string saved twice results in different ciphertexts (preventing frequency analysis).

**The Master Key:** The secret key used to encrypt the SharedPreferences is generated by the OS and stored physically inside the Android Keystore (Trusted Execution Environment / Titan M chip). The app never sees the raw bytes of the Master Key.

| Trade-offs of EncryptedSharedPreferences |
| :--- |
| **🟢 PRO:** Extreme security. Immune to rooted device data extraction (the key stays in hardware).
| **🟢 PRO:** Simple API drop-in replacement for standard SharedPreferences.
| **🔴 CON (Speed):** **Incredibly slow.** Reading/Writing takes 10x-50x longer than normal preferences because data must be piped down to the hardware Keystore chip for decryption.
| **🔴 CON (Size):** Ciphertexts take up significantly more disk space than plain text.
| **🛠️ Staff Advice:** NEVER put `EncryptedSharedPreferences` on the Main Thread. NEVER use it for massive datasets (like a list of 1000 POIs). Use it *only* for Session Tokens or PII.

### B. SQLCipher (Encrypted Room Database)
**How it works in detail:**
It replaces the default SQLite engine with one that performs 256-bit AES encryption of database files on the fly. You provide a passphrase when opening the DB. 
*Where does the passphrase come from?* You generate a random 256-bit byte array on first launch, encrypt that byte array using the Android Keystore, save the encrypted array in normal `SharedPreferences`, and use the decrypted array to open SQLCipher.

| Trade-offs of SQLCipher |
| :--- |
| **🟢 PRO:** Secures massive datasets offline (e.g., offline maps, user communication logs).
| **🔴 CON (APK Size):** Adds ~2-3MB to your APK because you are shipping an entire custom C++ database engine inside your app.
| **🔴 CON (Complexity):** Huge pain during database migrations. If the user clears app data but leaves the Keystore intact, or vice versa, the DB becomes a permanently unreadable brick and must be deleted.
| **🛠️ Staff Advice:** Do not use SQLCipher for public data (e.g., a list of public EV charging stations). Only encrypt databases containing PII or proprietary Harman algorithms.

## 2. Data in Transit (Network Security)

### A. Certificate Pinning
**How it works in detail:**
HTTPS ensures traffic is encrypted between the device and the server. However, if a user installs a malicious Root Certificate on their phone (via Charles Proxy, or malware), they can execute a Man-In-The-Middle (MITM) attack. 
Certificate Pinning tells the OkHttp client: *"Even if the Android OS says this certificate is valid, REJECT IT unless the Public Key hash matches this exact string: `sha256/7HIpactk...`"*.

| Trade-offs of Certificate Pinning |
| :--- |
| **🟢 PRO:** Mathematically guarantees you are talking to Harman's servers, defeating all MITM proxies or compromised Corporate WiFis.
| **🔴 CON (The "Time Bomb"):** Certificates expire (usually every 1-2 years). If your DevOps team rotates the server certificate and forgets to update the app, **every single user is instantly locked out of the app**.
| **🛠️ Staff Advice (The Fix):** Never pin the "Leaf" (bottom-level) certificate. Pin the Intermediate or Root CA (Certificate Authority) of your cloud provider (e.g., AWS or Let's Encrypt). ALWAYS include 2 backup pins in your `network_security_config.xml` that your DevOps team controls in cold storage.

## 3. Platform Integrity & Root Detection

### Play Integrity API (Formerly SafetyNet)
**How it works in detail:**
You cannot trust the client app. If the app checks `isRooted()`, a hacker can decompile the APK, change `return true` to `return false`, recompile it, and bypass your check.
Play Integrity relies on Google Play Services deeply embedded in the OS:
1. App requests a nonce (random string) from your Backend.
2. App passes the nonce to Google Play Services.
3. Google Play evaluates the hardware (Bootloader unlocked? Magisk installed? Emulated environment?).
4. Google returns a highly encrypted JWT to the App.
5. App forwards the JWT to your Backend. Your Backend decrypts it and sees if the device is genuine.

| Trade-offs of Play Integrity |
| :--- |
| **🟢 PRO:** Almost impossible to spoof, as the verification happens on completely separate Google servers, not inside your decompilable Kotlin code.
| **🔴 CON (Latency):** It requires a live network call to Google before the user can log in or perform a sensitive action. This adds 1-3 seconds of latency.
| **🔴 CON (AOSP/China):** Fails entirely on devices without Google Play Services (e.g., Huawei, Amazon Fire, or Chinese market vehicles).
| **🛠️ Staff Advice:** Do not check Play Integrity on every single API call. Check it during Login or during high-risk actions (Remote Engine Start), and cache the "trust" state in an encrypted session token.

## 4. Biometric Authentication (With Keystore Binding)

**How it works in detail:**
You don't just ask `BiometricPrompt` if the user is authenticated (because a rooted device can spoof the `onSuccess` callback). Instead, you ask the Android Keystore to generate an asymmetric key pair (RSA/EC) that is **cryptographically bound** to user authentication.
When you want to decrypt the user's Session Token, you pass the Cipher object to the `BiometricPrompt`. The Keystore chip will *refuse* to unlock the key to decrypt the token unless the fingerprint sensor hardware confirms a match.

| Trade-offs of Crypto-Bound Biometrics |
| :--- |
| **🟢 PRO:** Ultimate security. Even if the OS is completely compromised, the hacker cannot decrypt the session token because the hardware biometric chip holds the lock.
| **🔴 CON (UX Friction):** If the user adds a *new* fingerprint in Android Settings, the OS permanently invalidates your key! The user will be entirely logged out of your app and forced to type their password again.
| **🔴 CON (Fragmentation):** Android biometric hardware quality varies wildly across OEMs (Samsung vs Motorola vs cheap tablets). You must handle dozens of edge cases (`BIOMETRIC_ERROR_HW_UNAVAILABLE`).

## 5. Harman / Android Automotive OS (AAOS) Specific Security 🚗

Automotive security is critical because software vulnerabilities can lead to physical vehicle manipulation (e.g., unlocking doors, accelerating).

### A. IPC (Inter-Process Communication) Hardening via AIDL
**How it works:** In AAOS, apps (Spotify, Maps, Dashboard) run in isolated sandboxes. They communicate with the Vehicle Service via AIDL. 
**The Threat:** If a malicious weather app binds to your Vehicle Service, it could call `accelerateCar()` if you don't secure the AIDL endpoint.

| Trade-offs of Custom Permissions |
| :--- |
| **🟢 PRO:** `signature` level permissions ensure that only apps cryptographically signed by Harman or the Car Manufacturer can bind to high-risk AIDL services.
| **🔴 CON (Monolithic Coupling):** If you rely on `signature` permissions, third-party apps (like a parking app wanting to read the Odometer) cannot integrate with your system. You have to build complex OAuth-style delegation systems for third parties.

### B. SELinux (Security-Enhanced Linux) in AAOS
**How it works:** Android is built on Linux. Standard Linux uses Discretionary Access Control (DAC), where root (`su`) can do anything. AAOS enforces Mandatory Access Control (MAC) via SELinux. Every process has a strict "Context" (label). Even if an app achieves `root` execution, SELinux policies (written in C/C++ by Harman system engineers) will physically block the app from reading the `/dev/can` bus file interface.

| Trade-offs of SELinux Enforcing |
| :--- |
| **🟢 PRO:** Limits the "blast radius" of a zero-day exploit. A compromised browser app is trapped in its labeled sandbox.
| **🔴 CON (System Dev Slowdown):** Writing SELinux policies (`.te` files) is notoriously difficult. If an Android dev creates a new background service to read a new sensor, they will face `avc: denied` crashes until a system engineer updates the low-level OS policies.

## 6. Interview Summary (The "Why" Matrix)

If asked: "What security measures do you implement?", guide the answer through trade-offs:

1.  *"I use **EncryptedSharedPreferences** for tokens, but I'm careful to read from it asynchronously because it's slow due to hardware decoding."*
2.  *"I implement **Certificate Pinning**, but I pin intermediate CAs with backups to avoid apps bricking themselves when DevOps rotates certificates."*
3.  *"I integrate the **Play Integrity API** for anti-tampering, recognizing it adds network latency and fails on non-GMS devices (like some AAOS head units)."*
4.  *"I secure **AIDL bounds services** by strictly checking `Binder.getCallingUid()` to prevent Confused Deputy attacks from malicious apps installed on the head unit."*

## 7. Domain-Specific Security Scenarios 🏭

### A. Digital Gold & FinTech 🥇
**Q: A user's device is physically stolen while unlocked. How do you prevent the thief from transferring out all the user's Digital Gold?**
**A:** Enforce **Transaction-Level Biometric Authentication**. Do not simply rely on the app being "logged in."
1. When generating the RSA/EC signing key in the Android Keystore during login, use `setUserAuthenticationRequired(true)` and `setUserAuthenticationValidityDurationSeconds(-1)`.
2. This forces the Keystore to lock the transaction-signing key immediately after every use.
3. Every time the user presses "Transfer," you MUST present the `BiometricPrompt`. Only a successful scan will temporarily unlock the Keystore hardware, sign the transfer payload, and send it to the backend. The unlocked session does not persist.

### B. Insurance Claims 📄
**Q: An insurance app allows users to upload photos of car damage. How do you cryptographically prove to the backend that a photo was taken natively by the device camera right now, and not altered in Photoshop or downloaded from the internet?**
**A:** Use **Hardware-Backed Keystore Attestation & Scoped Storage**.
1. When the app takes the photo via CameraX, immediately sign the image hash (SHA-256) using a hardware-backed EC Key. 
2. Attach the generated Key Attestation Certificate chain (which proves the key actually resides in the Titan M chip and device is not rooted) to the upload payload.
3. The backend verifies the signature and the certificate chain. This guarantees the image hash was generated securely on a non-tampered device at the exact timestamp of the claim.

### C. BLE (Bluetooth Low Energy) Devices ⌚
**Q: Your app syncs proprietary data with a medical BLE band. A hacker runs a Bluetooth sniffer app on the same phone. How do you prevent them from intercepting the BLE GATT characteristics?**
**A:** Standard BLE pairing is easily intercepted by the Android OS or other apps with `BLUETOOTH_CONNECT` permissions.
1. Implement **Application-Layer Encryption (E2EE)**.
2. The BLE band and the App perform a Diffie-Hellman Key Exchange (e.g., ECDH) *over* the standard BLE connection upon first pairing. 
3. All subsequent GATT characteristic reads/writes consist of AES-GCM encrypted byte arrays. Even if a sniffer intercepts the BLE packets, they only see AES-encrypted garbage without the Session Key.

### D. Telemedicine & Healthcare 🩺
**Q: A telemedicine app stores downloaded PDF medical prescriptions on the device for offline viewing. How do you secure them so no other file manager app or malicious app can steal the PDFs?**
**A:** Never write medical data to external/shared storage (`Downloads/` or `Documents/`), even if it is a standard PDF.
1. Write the PDFs strictly to **Internal App-Specific Storage** (`context.filesDir`). Linux file permissions isolate this directory, making it impossible for other apps to read it on a non-rooted device.
2. If the user explicitly wants to save the PDF to their public `Downloads` folder, force them to invoke the **Storage Access Framework (SAF)** via an `Intent.ACTION_CREATE_DOCUMENT`. This clearly shifts the security responsibility to the user's explicit action and requires interaction with the system UI picker.
3. For extreme HIPAA compliance, encrypt the PDF bytes via `EncryptedFile` (Jetpack Security) before writing it to `context.filesDir`, ensuring it's unreadable even via physical memory extraction on a rooted device.
